# 1.1 Repo Bootstrap

## 1. Контекст

Новый проект `open-clodex-iflow` должен стартовать не как пустой каталог, а как Iron Dome-compatible workspace. Это означает, что до первой feature-реализации должны существовать docs, baseline, quality gates, task artifacts и базовые enforcement scripts.

Проект сам является инструментальным control plane, а не прикладным доменным сервисом. Поэтому bootstrap должен сосредоточиться на reproducible repo structure, docs-as-source-of-truth и smoke-проверках на обязательные артефакты.

## 2. Outcome

В репозитории существует минимальный, но содержательный starter kit, на котором можно безопасно продолжать реализацию provider discovery и orchestration logic.

## 3. Architecture integration

- module: `open_clodex_iflow`
- docs source of truth: `docs/*`
- enforcement surface: `enforcement/*`, `scripts/validate_story.py`, `Makefile`
- security: no secrets in repo, no implicit provider auth in docs or code

## 4. Step-by-step план

1. Создать root layout, docs, tasks и tests
2. Добавить smoke tests на scaffold completeness
3. Добавить minimal Python package и CLI skeleton
4. Добавить enforcement и story validation
5. Заполнить docs содержимым под продукт

## 5. Test cases

1. `test_cli_module_exists`
2. `test_package_init_exists`
3. `test_required_scaffold_files_exist`

## 6. Edge cases

- missing required docs
- missing package entrypoint
- missing story validation script
- accidental bootstrap without task artifacts

## 7. Acceptance criteria

- все required scaffold files существуют
- package skeleton присутствует
- docs описывают продукт, а не пустой template
- smoke tests проходят

## 8. Команды проверки

- `python -m pytest tests/unit/test_cli_smoke.py -q`
- `python -m pytest tests/integration/test_repo_scaffold.py -q`
- `make check-fast`

## 9. Что не делать

- не добавлять WSL-only runtime
- не строить plugin framework
- не тянуть web UI
