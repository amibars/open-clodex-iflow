# 1.3 Artifact Packet Contract

## 1. Контекст

`/solo` и `/orch` должны быть раздельны по privacy boundary. Это невозможно без четкого artifact packet contract, который определяет, что именно можно отправлять reviewer pool и что запрещено выносить из private mode.

## 2. Outcome

Существует ясный contract для `artifact.json`, `review.json` и `consolidated_review.json`, который код и docs используют одинаково.

## 3. Architecture integration

- docs: `docs/ARCH_SPEC.md`, `docs/ORCHESTRATED.md`
- code: future `src/open_clodex_iflow/orchestration/*`
- review integration: `Claude`, `iFlow`, `OpenCode` adapters

## 4. Step-by-step план

1. Описать canonical packet schema в docs
2. Реализовать dataclasses/typed structures
3. Добавить validator tests
4. Встроить privacy separation for `/solo`

## 5. Test cases

1. artifact packet contains required fields
2. solo packet builder refuses external review payload
3. consolidated review groups findings predictably
4. invalid packet shape fails validation

## 6. Edge cases

- empty findings list
- conflicting verdicts from reviewers
- partial reviewer failure
- missing next step plan from Codex artifact

## 7. Acceptance criteria

- contract documented and testable
- `/solo` and `/orch` boundaries explicit
- consolidated verdict supports proceed/fix_code/fix_plan/block

## 8. Команды проверки

- `python -m pytest tests/unit -q`
- `make enforce`

## 9. Что не делать

- не передавать историю чата целиком
- не делать debate loop в v1
- не логировать приватный `/solo` контекст во внешние provider outputs
