# 1.2 Provider Discovery

## 1. Контекст

UX проекта опирается на reuse системных CLI и уже существующего auth/session state. Поэтому первая runtime capability после bootstrap — надежно обнаруживать binaries и известные state dirs для `codex`, `claude`, `iflow`, `opencode`.

## 2. Outcome

Команда `doctor` и discovery layer умеют перечислить доступные provider binaries и локальные state directories без вывода секретов.

## 3. Architecture integration

- files: `src/open_clodex_iflow/cli.py`, future `src/open_clodex_iflow/adapters/*`
- contract: provider snapshot data structure
- security: only metadata, no token dumping

## 4. Step-by-step план

1. Выделить discovery helpers из CLI слоя
2. Нормализовать provider snapshot schema
3. Написать unit tests на known-path detection
4. Расширить `doctor` human-readable and JSON output

## 5. Test cases

1. detect binary in PATH
2. detect existing state dir
3. omit missing dirs from output
4. never include secret contents

## 6. Edge cases

- binary exists, state dir missing
- state dir exists, binary missing
- provider has multiple candidate state dirs
- path contains Windows-specific separators

## 7. Acceptance criteria

- provider discovery works without crashing on missing tools
- output includes only metadata
- `doctor` can be used as preflight step before `/orch`

## 8. Команды проверки

- `python -m pytest tests/unit -q`
- `python -m open_clodex_iflow.cli doctor`

## 9. Что не делать

- не читать токены из файлов
- не инициировать новый login flow
- не смешивать discovery с orchestration decisions
